Security Context\n\n### 1.1 What This System Protects\n\nHealthGate is a centralized authentication platform for all Google Health web applications. It is architecturally classified as **Zone 1 (Auth Plane)** within a 3-Zone isolation model. HealthGate protects and manages the following categories of security-critical data:\n\n**Data Assets Under Protection:**\n\n| Asset Class | Examples | Sensitivity Level |\n|---|---|---|\n| Authentication Credentials | Hashed passwords (Argon2id), TOTP secrets (AES-256 encrypted), WebAuthn public keys, recovery code hashes (bcrypt, cost 12) | CRITICAL |\n| Session Material | Keycloak session identifiers, access tokens (JWT/RS256), refresh tokens, PKCE code verifiers, CSRF tokens | CRITICAL |\n| Identity Metadata | User UUIDs, email addresses, full names, role assignments, MFA enrollment status | HIGH |\n| Device Trust Artifacts | Trusted device cookies, device fingerprint hashes, user-agent/IP association history | HIGH |\n| Audit Records | Structured JSON event logs for all authentication events, admin actions, policy changes | HIGH (integrity-critical) |\n| Configuration State | Realm policies, client registrations, SAML/OIDC metadata, password policy parameters, lockout thresholds | HIGH |\n\n**What HealthGate Explicitly Does NOT Protect or Store:**\n\n- Protected Health Information (PHI) -- zero PHI resides in Zone 1 by architectural constraint\n- Patient diagnoses, medications, lab results, treatment records\n- Any of the 18 HIPAA identifiers beyond name and email (which are necessary for authentication)\n- Social Security Numbers, insurance information, or financial data\n\nThis separation is the foundational security guarantee: if an attacker fully compromises Zone 1, they obtain authentication material but gain zero access to Protected Health Information stored in Zone 3.\n\n### 1.2 Trust Boundaries (3-Zone Model)\n\n```\n UNTRUSTED                    ZONE 1                        ZONE 2                      ZONE 3\n (Internet) Zone 1 is the only zone with internet-facing endpoints. All user-facing input validation occurs here. Security headers are applied at this boundary.\n2. **TB2 (Zone 1 to Zone 2):** Communication is exclusively via mTLS. Zone 2 receives only signed JWTs from Zone 1 -- never raw credentials. Zone 2 validates tokens using cached JWKS and enforces rate limits.\n3. **TB3 (Zone 2 to Zone 3):** Communication is exclusively via mTLS. Zone 3 services receive only validated requests with verified role claims. Zone 3 performs its own authorization decisions based on JWT claims.\n\n**Critical Invariant:** No credential material (passwords, TOTP secrets, recovery codes, session tokens) ever crosses TB2 or TB3. Tokens crossing TB2 are signed, time-limited, and audience-restricted.\n\n### 1.3 Regulatory Context\n\n**HIPAA Security Rule (45 CFR Part 164, Subpart C):**\nHealthGate is a component of a covered entity's information system that creates, receives, maintains, or transmits electronic protected health information (ePHI) -- specifically, it controls access to systems that contain ePHI. While HealthGate itself stores zero PHI, it is squarely within scope of the HIPAA Security Rule because it is the access control mechanism for systems that do contain ePHI.\n\n**2025 HIPAA Security Rule NPRM (Notice of Proposed Rulemaking):**\nThe 2025 NPRM represents the first major update to the HIPAA Security Rule in 20 years. Key changes relevant to HealthGate:\n\n1. **MFA becomes explicitly mandatory** -- the \"addressable\" ambiguity is eliminated. HealthGate's mandatory MFA enrollment (FR6) and enforcement (FR5) directly satisfy this requirement.\n2. **Encryption becomes explicitly required** (no longer \"addressable\") -- TLS 1.3 in transit and AES-256 at rest are now non-negotiable.\n3. **72-hour incident reporting** -- requires HealthGate to support rapid forensic analysis via audit logs.\n4. **Technology asset inventory** -- requires documentation of all authentication components and their security configurations.\n5. **180-day compliance window** from rule finalization -- HealthGate's launch timeline must align with this deadline.\n6. **Patch management within 15 days for critical vulnerabilities** -- affects Keycloak, Node.js, and PostgreSQL patch cadence.\n7. **Annual penetration testing** explicitly required -- HealthGate's quarterly schedule exceeds this minimum.\n8. **Network segmentation** explicitly required -- HealthGate's 3-Zone model satisfies this by design.\n\n---\n\n## 2. STRIDE Threat Model\n\n### 2.1 Spoofing\n\n| Threat ID | Threat | Attack Vector | Likelihood | Impact | Mitigation | Residual Risk |\n|---|---|---|---|---|---|---|\n| S-1 | Credential Theft via Phishing | Attacker sends emails mimicking HealthGate login, harvests credentials on a lookalike domain | HIGH | CRITICAL | (1) Mandatory MFA (TOTP/WebAuthn) ensures stolen password alone is insufficient. (2) HSTS preload prevents SSL stripping on legitimate domain. (3) CSP `form-action 'self' {keycloak-url}` prevents form submission to unauthorized domains. (4) New device notification (FR22) alerts users to unauthorized access. (2) TLS 1.3 prevents network interception. (3) CSP with strict `script-src 'self'` prevents XSS payload execution. (4) Access tokens are 5-minute lived -- limiting window of exploitation. (5) Session binding to IP/User-Agent with anomaly detection. | LOW |\n| S-3 | Token Replay Attack | Attacker captures a valid JWT and replays it to access protected resources | MEDIUM | HIGH | (1) Access tokens expire in 5 minutes (`exp` claim enforced). (4) Replay protection via `NotOnOrAfter` condition and `InResponseTo` correlation. (5) Audience restriction enforced (`<AudienceRestriction>`). | VERY LOW |\n| S-5 | Credential Stuffing (using breach databases) | Attacker uses credentials from external breaches to attempt login | HIGH | HIGH | (1) Password breach list check via Have I Been Pwned API (k-anonymity model) at registration and password change (FR9). (2) Brute-force protection: 5-attempt lockout, progressive doubling (FR12). (3) IP-rate limiting: 20 failed attempts per IP per 5 minutes. (4) Mandatory MFA renders password-only access impossible. (5) New device notification (FR22) for unfamiliar logins. | LOW |\n\n### 2.2 Tampering\n\n| Threat ID | Threat | Attack Vector | Likelihood | Impact | Mitigation | Residual Risk |\n|---|---|---|---|---|---|---|\n| T-1 | JWT Claim Modification | Attacker modifies JWT claims (e.g., role escalation) in transit or at rest | LOW | CRITICAL | (1) JWTs signed with RS256 (RSA 2048-bit). Security Headers Specification\n\nAll headers are applied at the Next.js middleware layer (`middleware.ts`) and are enforced on every HTTP response. These values are non-configurable by downstream applications.\n\n### 6.1 Content-Security-Policy\n\n```\nContent-Security-Policy:\n  default-src 'self';\n  script-src 'self';\n  style-src 'self' 'unsafe-inline';\n  img-src 'self' data:;\n  font-src 'self' https://fonts.gstatic.com https://api.fontshare.com;\n  connect-src 'self' https://auth.googlehealth.com;\n  frame-ancestors 'none';\n  frame-src 'none';\n  form-action 'self' https://auth.googlehealth.com;\n  base-uri 'self';\n  object-src 'none';\n  media-src 'none';\n  worker-src 'self';\n  manifest-src 'self';\n  upgrade-insecure-requests;\n  block-all-mixed-content\n```\n\n**Directive Rationale:**\n- `script-src 'self'` -- Only first-party scripts. No `unsafe-eval`, no `unsafe-inline` for scripts. If inline scripts are needed for Next.js hydration, use `nonce`-based CSP with per-request random nonces.\n- `style-src 'self' 'unsafe-inline'` -- `unsafe-inline` required for Tailwind CSS and styled-jsx. Consider migrating to nonce-based styles in Phase 2.\n- `img-src 'self' data:` -- `data:` required for QR code rendering (MFA setup page, FR6).\n- `font-src` includes Google Fonts (Cormorant Garamond, Lora) and Fontshare (Satoshi) per UX spec.\n- `connect-src 'self' https://auth.googlehealth.com` -- Allows XHR/fetch to HealthGate BFF and Keycloak.\n- `frame-ancestors 'none'` -- Prevents HealthGate from being embedded in any iframe (anti-clickjacking).\n- `form-action 'self' https://auth.googlehealth.com` -- Restricts form submission to HealthGate and Keycloak only.\n- `object-src 'none'` -- Blocks Flash, Java applets, and other plugin content.\n- `upgrade-insecure-requests` -- Forces subresource requests to HTTPS.\n- `block-all-mixed-content` -- Prevents any mixed HTTP/HTTPS content.\n\n### 6.2 Other Security Headers\n\n| Header | Value | Notes |\n|---|---|---|\n| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains; preload` | 1-year HSTS. Includes all subdomains. Submit to HSTS preload list (hstspreload.org). |\n| `X-Frame-Options` | `DENY` | Legacy clickjacking protection. Redundant with CSP `frame-ancestors 'none'` but included for older browser compatibility. |\n| `X-Content-Type-Options` | `nosniff` | Prevents MIME type sniffing. Browser must respect declared Content-Type. |\n| `Referrer-Policy` | `strict-origin-when-cross-origin` | Same-origin requests: full URL. Cross-origin requests: origin only. HTTPS-to-HTTP: no referrer. Prevents token/path leakage in referrer header. |\n| `Permissions-Policy` | `camera=(), microphone=(), geolocation=(), payment=(), usb=(), magnetometer=(), gyroscope=(), accelerometer=(), ambient-light-sensor=(), autoplay=(), encrypted-media=(), fullscreen=(self), picture-in-picture=()` | Disables all unnecessary browser APIs. Only `fullscreen` permitted for the application itself. |\n| `Cache-Control` | `no-store, no-cache, must-revalidate, private` | Applied to all authentication pages (login, register, MFA, password reset). Prevents caching of authentication state in browser or proxy caches. |\n| `Pragma` | `no-cache` | Legacy cache prevention for HTTP/1.0 proxies. |\n| `X-Request-Id` | `{uuid-v4}` | Unique per-request identifier for tracing. Generated by Next.js middleware. Forwarded to Keycloak and logged in audit events as `correlation_id`. |\n| `X-DNS-Prefetch-Control` | `off` | Disables DNS prefetching to prevent information leakage about linked resources. |\n| `Cross-Origin-Opener-Policy` | `same-origin` | Isolates browsing context to prevent cross-origin attacks (Spectre mitigation). |\n| `Cross-Origin-Resource-Policy` | `same-origin` | Prevents resources from being loaded by cross-origin pages. |\n| `Cross-Origin-Embedder-Policy` | `require-corp` | Ensures all loaded resources have appropriate CORS/CORP headers. |\n\n### 6.3 Cookie Security Attributes\n\nAll cookies set by HealthGate use the following attributes:\n\n```\nSet-Cookie: hg_access={encrypted_token};\n  Path=/;\n  HttpOnly;\n  Secure;\n  SameSite=Strict;\n  Max-Age=300;\n  Domain=.googlehealth.com\n\nSet-Cookie: hg_refresh={encrypted_token};\n  Path=/api/auth/refresh;\n  HttpOnly;\n  Secure;\n  SameSite=Strict;\n  Max-Age={realm_max_session_seconds};\n  Domain=.googlehealth.com\n\nSet-Cookie: hg_csrf={csrf_token};\n  Path=/;\n  Secure;\n  SameSite=Strict;\n  Max-Age=86400;\n  Domain=.googlehealth.com\n  (Note: NOT HttpOnly -- must be readable by JavaScript for double-submit pattern)\n\nSet-Cookie: hg_device_trust={encrypted_device_hash};\n  Path=/;\n  HttpOnly;\n  Secure;\n  SameSite=Strict;\n  Max-Age=2592000;\n  Domain=.googlehealth.com\n  (Note: 30-day lifetime per FR21)\n```\n\n---\n\n## 7.
